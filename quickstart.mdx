---
title: "Get started with CozoDB"
description: "Write your first CozoDB query in the browser — no installation needed. Then create a relation, insert data, and explore graph traversal with real examples."
---

The fastest way to try CozoDB is in your browser: a full CozoDB instance runs in WASM at near-native speed, so you can learn the query language before installing anything. Once you're ready, follow the steps below to move to an embedded or server deployment.

<Steps>
  <Step title="Open the browser demo">
    Navigate to [https://www.cozodb.org/wasm-demo/](https://www.cozodb.org/wasm-demo/) and open the interactive editor. This runs a complete, in-memory CozoDB instance using WebAssembly — no account or installation required.
  </Step>

  <Step title="Run your first query">
    Type the following into the editor and run it:

    ```datalog
    ?[a] := a in [1, 2, 3]
    ```

    CozoDB returns a relation with one column (`a`) and three rows. Breaking this down:

    - `?[a]` — the **head** of the rule. `?` is the special name for the result relation; `a` is the column you want returned.
    - `:=` — separates the head from the **body**.
    - `a in [1, 2, 3]` — constrains `a` to each element of the list in turn.

    Every CozoDB query is a Datalog rule. More complex queries simply add more atoms to the body or chain multiple rules together.
  </Step>

  <Step title="Explore graph queries with the airport dataset">
    The WASM demo ships with a `*route` relation containing two columns — `fr` (from) and `to` — representing flight routes between airports. `FRA` is the IATA code for Frankfurt Airport. These three queries show progressively more powerful graph traversal:

    **Direct connections** — how many airports are reachable from FRA in one hop?

    ```datalog
    ?[count_unique(to)] := *route{fr: 'FRA', to}
    ```

    | count_unique(to) |
    |------------------|
    | 310              |

    **One-stop connections** — how many airports are reachable with exactly one layover?

    ```datalog
    ?[count_unique(to)] := *route{fr: 'FRA', to: stop},
                           *route{fr: stop, to}
    ```

    | count_unique(to) |
    |------------------|
    | 2222             |

    **Recursive reachability** — how many airports are reachable by any number of stops? This uses a recursive rule: `reachable` refers to itself.

    ```datalog
    reachable[to] := *route{fr: 'FRA', to}
    reachable[to] := reachable[stop], *route{fr: stop, to}
    ?[count_unique(to)] := reachable[to]
    ```

    | count_unique(to) |
    |------------------|
    | 3462             |

    <Tip>
    Recursion in CozoScript is declarative — you define _what_ is reachable, not _how_ to traverse the graph. The query engine handles cycle detection and termination automatically.
    </Tip>
  </Step>

  <Step title="Create a relation and insert data">
    Relations in CozoDB have an explicit schema. Create a `user` relation with `id` as the key column and `name`/`email` as value columns:

    ```datalog
    :create user {id: Int => name: String, email: String}
    ```

    The `=>` separates **key columns** (left) from **value columns** (right). Key columns form the primary key.

    Insert a row using a `:put` mutation:

    ```datalog
    ?[id, name, email] <- [[1, 'Alice', 'alice@example.com']]
    :put user {id => name, email}
    ```

    The `<-` operator provides inline data as a list of tuples. `:put` inserts or upserts each row.

    Insert a second row:

    ```datalog
    ?[id, name, email] <- [[2, 'Bob', 'bob@example.com']]
    :put user {id => name, email}
    ```
  </Step>

  <Step title="Query the relation">
    Read all rows back from the `user` relation. The `*` prefix on a relation name means "scan this stored relation":

    ```datalog
    ?[name, email] := *user{name, email}
    ```

    | name  | email                 |
    |-------|-----------------------|
    | Alice | alice@example.com     |
    | Bob   | bob@example.com       |

    You can filter inline using unification:

    ```datalog
    ?[name, email] := *user{id: 1, name, email}
    ```

    | name  | email             |
    |-------|-------------------|
    | Alice | alice@example.com |
  </Step>
</Steps>
